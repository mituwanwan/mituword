# 文档版本控制规范

> **版本**: v1.0  
> **创建日期**: 2026-05-14  
> **适用范围**: 个人网站系统项目文档

---

## 1. 版本号规范

### 1.1 版本号格式

采用语义化版本控制 (Semantic Versioning)：

```
主版本号.次版本号.修订号
例如: v1.2.3
```

### 1.2 版本号变更规则

| 版本类型 | 变更时机 | 示例 |
|---------|---------|------|
| **主版本号 (Major)** | 文档体系重大重构、结构变更、或与其他系统合并时 | v1.x.x → v2.0.0 |
| **次版本号 (Minor)** | 新增文档、新增章节、或功能模块文档增加时 | v1.1.x → v1.2.0 |
| **修订号 (Patch)** | 错误修正、内容更新、链接修复、格式调整时 | v1.1.1 → v1.1.2 |

### 1.3 预发布版本

开发中的文档版本使用预发布标识：

```
v1.2.0-alpha.1  # 内部评审版本
v1.2.0-beta.1   # 团队评审版本
v1.2.0-rc.1     # 发布候选版本
```

## 2. 文档状态标识

### 2.1 状态类型

| 状态 | 标识 | 说明 | 颜色标记 |
|------|------|------|---------|
| 草稿 | `DRAFT` | 正在编写中，尚未完成 | 🟡 黄色 |
| 评审中 | `REVIEW` | 已完成编写，等待评审 | 🔵 蓝色 |
| 已批准 | `APPROVED` | 已通过评审，可以发布 | 🟢 绿色 |
| 已发布 | `PUBLISHED` | 已正式发布，供团队使用 | 🟢 绿色 |
| 已归档 | `ARCHIVED` | 已过时，保留供参考 | ⚪ 灰色 |
| 已废弃 | `DEPRECATED` | 不再使用，将被删除 | 🔴 红色 |

### 2.2 状态转换流程

```
[DRAFT] → [REVIEW] → [APPROVED] → [PUBLISHED]
                              ↓
                        [ARCHIVED] (过时)
                              ↓
                        [DEPRECATED] (废弃)
```

## 3. 变更记录规范

### 3.1 变更记录格式

每份文档必须包含变更记录表格：

```markdown
## 文档变更记录

| 版本 | 日期 | 变更内容 | 作者 | 评审人 |
|------|------|---------|------|--------|
| v1.1.0 | 2026-05-15 | 新增API限流策略章节 | 张三 | 李四 |
| v1.0.1 | 2026-05-14 | 修正数据库字段类型错误 | 王五 | - |
| v1.0.0 | 2026-05-14 | 初始版本 | 系统架构师 | - |
```

### 3.2 变更内容描述规范

- 使用简洁的动词开头：新增、修改、删除、修正、优化、重构
- 描述具体变更点，避免模糊表述
- 涉及多个变更时，使用分号分隔

**示例**:
- ✅ 正确: "新增GitHub同步API的速率限制处理章节"
- ❌ 错误: "更新了一些内容"

## 4. 文档命名规范

### 4.1 文件命名规则

```
[序号]-[文档类型]/[序号]-[文档名称].md

示例:
01-SDD/01-Software-Design-Document.md
02-API-Spec/01-API-Specification.md
```

### 4.2 命名约束

- 使用英文命名，单词间用连字符 `-` 分隔
- 序号使用两位数字，不足补零
- 文档类型使用大写字母缩写
- 避免使用特殊字符和空格

## 5. 文档评审流程

### 5.1 评审角色

| 角色 | 职责 |
|------|------|
| **作者** | 编写文档，发起评审请求 |
| **评审人** | 审查文档内容，提出修改意见 |
| **批准人** | 最终确认文档质量，批准发布 |

### 5.2 评审检查清单

- [ ] 内容准确性：技术细节是否正确
- [ ] 完整性：是否覆盖所有必要信息
- [ ] 一致性：与相关文档是否保持一致
- [ ] 可读性：语言是否清晰易懂
- [ ] 格式规范：是否符合文档模板要求
- [ ] 链接有效性：内部链接是否正确
- [ ] 版本更新：变更记录是否已更新

### 5.3 评审流程

```
1. 作者完成文档编写
   ↓
2. 作者更新文档状态为 [REVIEW]
   ↓
3. 作者指定评审人，发起评审请求
   ↓
4. 评审人审查文档，提出意见（3个工作日内）
   ↓
5. 作者根据意见修改文档
   ↓
6. 评审人确认修改，批准文档
   ↓
7. 作者更新状态为 [APPROVED] 或 [PUBLISHED]
```

## 6. 文档归档策略

### 6.1 归档条件

文档满足以下条件之一时应当归档：

- 文档内容已过时，被新文档替代
- 项目功能已下线，相关文档不再使用
- 文档版本已迭代超过3个主版本

### 6.2 归档操作

1. 在文档头部添加归档标识
2. 更新文档状态为 `ARCHIVED`
3. 在变更记录中注明归档原因
4. 在索引文档中标注为已归档
5. 保留文档文件，不移除

**归档标识示例**:

```markdown
> **⚠️ 已归档文档**
> 
> 本文档已于 2026-06-01 归档。
> 
> 替代文档: [新文档路径](新文档链接)
> 归档原因: 系统架构已升级，本文档不再适用
```

## 7. 版本控制工具集成

### 7.1 Git集成

所有文档变更必须通过Git进行管理：

```bash
# 文档变更提交规范
git add docs/
git commit -m "docs: 更新API规范，新增限流策略

- 新增速率限制章节
- 更新错误码定义
- 修正部分接口描述

相关文档: 02-API-Spec/01-API-Specification.md"
```

### 7.2 提交信息规范

文档相关的Git提交使用 `docs:` 前缀：

| 类型 | 说明 | 示例 |
|------|------|------|
| `docs:` | 文档内容更新 | `docs: 更新数据库设计，新增索引` |
| `docs(add):` | 新增文档 | `docs(add): 新增部署指南` |
| `docs(fix):` | 修正文档错误 | `docs(fix): 修正API路径错误` |
| `docs(arch):` | 归档文档 | `docs(arch): 归档旧版部署文档` |
